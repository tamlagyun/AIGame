# 《鲫鱼吃鲤鱼》游戏需求基线

需求版本：REQ-0.2
状态：已确认  
基线日期：2026-07-13

## 产品定位

游戏是一款横屏 2D 海洋大世界单机 PvE 游戏。玩家初始控制一条小鲫鱼，在有限但较大的海域中探索，与 AI 鱼类相互攻击并成长。

## 核心玩法

- 玩家使用虚拟摇杆进行二维游动。
- 初始普通攻击为面向当前朝向的短距离撕咬。
- 初始技能为带冷却的冲刺撕咬。
- 不同技能对应不同撕咬动作标识，供后续动画系统绑定。

## 工具复用

- 通用图片处理能力必须放置在工作区 `Tools/image-processing`，不得写死《鲫鱼吃鲤鱼》的目录、资源名称或生成文件名。
- 项目内只保留项目配置、资源登记和明确的导入参数；通用工具不得直接修改项目资源，除非调用时明确传入项目内输出路径。
- 击败其他鱼后获得经验；升级提高生命上限并回复部分生命。
- 其他鱼由 AI 控制，可攻击玩家或其他鱼。

## 世界与镜头

- 首版采用有限大海域，后续可升级为分区流式加载。
- 使用位于水中的 2D 正交镜头，从海底正上方垂直向下俯视；地图不得出现水面、天空、地平线或侧视海底剖面。
- 镜头跟随玩家在海底平面内移动。
- 鱼身体使用圆形或胶囊形轻量命中判定，核心结算不依赖完整刚体模拟。

## UI 与输入

- 横屏设计分辨率为 1280 × 720。
- 左侧预留虚拟摇杆，右侧预留普通攻击和初始技能按钮。
- UI 必须适配安全区、刘海屏、平板和折叠屏。
- 正式 UI 和美术禁止使用 SVG 或程序化图形拼凑。
- 所有 UI 修改必须遵守项目 `AGENTS.md` 的 UI 操作前门禁、坐标分层、EditBox 和 IDE/Web 双端验收规则；自动测试不得替代目标运行环境的视觉与交互验收。
- Canvas、WorldRoot、HudRoot 与游戏相机必须采用统一中心基准；不得通过世界根和 HUD 根同时增加相反固定偏移来让单一运行环境的结果碰巧对齐。

## 平台范围

- 微信小游戏
- 抖音小游戏
- Android
- iOS
- HarmonyOS

本阶段不接入平台 SDK，只建立统一接口和构建入口。正式导出仍需要各平台 SDK、AppID、签名和工具链；iOS 原生工程需要 macOS。

## 性能基线

- 主流设备目标 60 FPS，中低端设备不得低于 30 FPS。
- 后续实现必须使用对象池、视野外休眠和分区激活控制鱼群成本。
- 正式资源应使用图集和平台适配纹理，小游戏资源需支持首包与远程包拆分。
- 第一版性能预算暂定：最多 30 条活动鱼、屏幕内最多 16 条完整更新鱼；实际数值须通过真机压测修订。

## 本阶段范围

包含工程骨架、场景层级、核心类型、配置样例、存档契约、平台接口、构建入口、测试、需求和美术档案。

不包含多人战斗结算、商业 SDK、签名安装包和生产数据库；多人可见与同步移动由 REQ-0.5 覆盖。

## REQ-0.5 多人联机移动

- 服务端采用 Node.js 20、TypeScript、Fastify 和 WebSocket。
- 默认海域匹配后进入单房间，最多 50 名玩家。
- 客户端发送移动输入，服务器以 20Hz 校验位置和旋转并以 10Hz 广播快照。
- 客户端支持本地预测和服务器校正，远端玩家使用快照插值显示。
- 测试版使用内存账号和房间；断线玩家立即退出房间。
- 本阶段不结算普通撕咬、冲刺撕咬、伤害、死亡或复活。

## REQ-0.2 初始化地图与玩家移动

- `MainScene` 显示首张正式海洋地图背景和玩家小鲫鱼。
- 玩家小鲫鱼使用 6 帧透明 PNG 循环游泳动画。
- 桌面预览支持 WASD 和方向键；移动端及小游戏支持屏幕左侧拖动式虚拟摇杆输入。
- 玩家可以在 3840 × 2160 的首张海域内自由游动，不得越过地图边界。
- 玩家仍可向平面内任意方向移动，但鱼儿美术朝向仅保留左、右两种；上下移动和停止时维持最近一次水平朝向，不允许鱼身随移动方向上下或斜向旋转。
- 鱼儿逻辑方向固定为 `0°=向右、180°=向左`；游泳、攻击等原图实际方向由鱼配置登记，并在加载时归一到 `artFacingDirection`。方向错误只能通过配置修正，不得分别修改移动、攻击或联机同步逻辑。
- 镜头平滑跟随玩家，并在地图边缘停止继续移动。
- 本阶段的虚拟摇杆只实现交互区域和文字提示；正式摇杆 UI 需另行确认美术提示词后生成。

## REQ-0.3 地图视角修订

- 首张地图改为水中垂直向下俯视海底的平面 2D 视角。
- 海底保持开阔，礁石、珊瑚和水草主要分布于边缘，中央区域供玩家自由游动。
- 禁止水面、天空、地平线、侧视海底剖面和斜俯视透视构图。
## REQ-0.4 UI 操作骨架

- 横屏安全区提供左下虚拟摇杆，支持拖动方向并驱动玩家移动。
- 下方偏中间提供普通撕咬按钮，右侧提供冲刺撕咬技能按钮。
- 普通撕咬具备 0.55 秒冷却并生成朝向扇形攻击反馈；冲刺撕咬具备 5 秒冷却，沿鱼儿朝向突进 240 世界单位并生成冲刺/撕咬反馈。伤害、命中和多人战斗结算仍由后续 CombatSystem 接入。

## REQ-0.7 受击动作表现

- 服务器确认命中后，主动攻击鱼播放现有撕咬、咬合和甩头撕扯动画，被攻击鱼同步切换为独立受击序列帧。
- 受击表现只替换美术帧，不修改被攻击鱼的世界位置、服务器位置、移动状态或网络快照坐标，不增加击退、吸附、抓取或控制效果。
- 受击动画必须与攻击动画的咬合节奏对应，通过鱼体局部受力弯曲、表情、鳍尾惯性和上下甩动姿态表现被咬住与甩扯；每帧角色中心、锚点、尺寸和画格位置保持一致。
- 咬合与甩扯的核心帧中，受击鱼应翻转为白色肚皮朝向镜头上方的“翻肚”姿态，再在松开后恢复正常游姿；全程保持身体完整，不出现血液、伤口、裂口或恐怖表现。
- 本地玩家和远端玩家收到同一服务器伤害事件后播放相同受击动画；死亡时由受击动画衔接死亡表现。

## REQ-0.8 鱼儿头顶生命条 UI

- 每名本地与远端玩家鱼儿都必须在头顶中上方显示生命条，生命条属于独立 HUD Overlay，不得作为鱼节点子节点。
- 生命条位置必须由鱼儿世界坐标转换为 `FishHealthOverlay` 局部坐标，并在鱼儿移动、镜头移动、左右翻转、攻击、受击、死亡和复活时持续跟随且保持屏幕正向。
- 生命条使用正式位图边框、空槽与可水平裁剪的生命填充；禁止使用 SVG、`Graphics` 或纯色临时块拼凑正式效果。
- 生命填充比例使用 `health / maxHealth`，从左向右显示剩余生命，扣血时右端随比例缩短，生命为零时填充完全消失。
- 当前生命值与最大生命值以 `当前/最大` 的数字格式居中叠放在血条最前层，数字不得被边框或填充遮挡。
- 所有玩家复用同一血条组件结构和更新逻辑；服务器状态是多人生命值的最终来源，本地单机模式保留安全默认值。
- IDE 内嵌预览与 Web 浏览器预览共用同一套 Cocos `Sprite + Label` 实现，并分别进行视觉验收。

## REQ-0.9 多人动作同步

- 普通撕咬和冲刺撕咬由服务器生成递增动作序号，并在 `skillEffect` 与后续状态快照中携带当前动作。
- 每个客户端对同一动作序号只播放一次；远端鱼即使错过瞬时技能事件，也能从状态快照补偿播放对应动作。
- 远端动作只替换角色 SpriteFrame，不修改远端鱼的服务器位置、世界坐标或朝向规则。

## REQ-0.10 远端鱼儿待机动画

- 每名远端鱼儿创建后必须循环播放游泳待机序列帧，不能停留在单张初始帧。
- 远端撕咬、受击和死亡动作期间暂停待机循环；动作结束或复活后恢复游泳循环。
- 待机动画只更新 SpriteFrame，不修改服务器位置、网络快照坐标、朝向或生命条坐标转换。

## REQ-0.11 右下技能扇形布局

- 右下技能区域采用移动 MOBA 常见的扇形交互结构，但不复制具体游戏的美术资源。
- 普攻按钮作为右下技能区的主按钮和圆心，其它技能从普攻左侧沿圆弧向上排列。
- `ActionControlsRoot` 作为技能区域唯一整体容器，通过安全区 Widget 右下对齐；所有按钮内部只使用容器局部坐标。
- 技能区包含四个技能按钮：原冲刺撕咬占据最左下第一个圆弧槽位，新增三个技能依次沿圆弧向上排列。
- 新增三个技能在原型验证阶段复用现有冲刺技能图标，并调用相同的冲刺撕咬技能效果、服务端技能 ID 和共享冷却；用户确认布局后再生成独立图标并实现独立技能效果。
- 普攻和技能名称显示在对应图标内部下侧，文字下边缘与图标下边缘对齐，并使用高对比描边保证可读性；禁止把名称放在图标外侧。
- 四个技能进入共享冷却时，图标上覆盖半透明深色径向蒙板；蒙板从 12 点方向开始顺时针转动一圈并随剩余冷却持续减少，冷却结束时完全消失。
- 冷却蒙板复用当前正式技能位图进行染色和径向裁剪，不新增 SVG、Graphics 或重复 PNG；技能名称显示在蒙板上层。
- 每个按钮的正式图像、触摸区域和文字标签必须属于同一个按钮节点，不得发生显示位置与事件位置分离。

## REQ-0.12 鲸吞技能

- 第二个圆弧技能槽位由临时技能改为正式技能“鲸吞”，临时采用 `ART-PROMPT-UI-0007 v1` 生成并登记的独立 PNG 图标；只替换该按钮的图片来源，不改变 `ActionControlsRoot`、按钮位置、尺寸、触摸区域、标签或冷却蒙板结构。该图标因含牙齿与原提示词排除项冲突，属于用户明确指定的暂用资源，后续应由已存档待确认的 v2 替换。
- 鲸吞由服务器在施法者 800 世界单位范围内选择最近的存活、非无敌敌方玩家；客户端不得提交或伪造目标。范围内没有有效目标时，服务器返回 `noTarget`，客户端取消本次冷却。
- 服务器确认目标后，将施法者的权威位置瞬移到目标当前权威位置，并向房间内所有客户端同步施法者、目标、位置和 3 秒效果时长。
- 鲸吞不造成额外伤害：施法者播放现有撕咬甩扯攻击动作，身体放大为正常尺寸的 3 倍并变为 50% 透明；被选中目标只变为 50% 透明。
- 3 秒后双方透明度恢复 100%，施法者恢复正常尺寸；目标的服务器坐标、生命值和动作状态不因该视觉效果改变。
- 鲸吞使用独立 8 秒冷却和独立径向蒙板；冲刺、临时技能 3、临时技能 4 继续共享原冲刺技能的 5 秒冷却。
- `skillEffect` 与动作状态快照携带 `targetId`、`effectDurationMs`/`actionRemainingMs`，保证晚加入或漏收瞬时事件的客户端能够补偿显示。
