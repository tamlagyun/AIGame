# 《鲫鱼吃鲤鱼》UI 布局基线

## 目标

本阶段采用横屏 1280×720 的安全区布局，交互结构参考移动 MOBA 的成熟布局逻辑，但不复制任何具体游戏的美术资源。

## 控件位置

- `JoystickControlRoot`：整体通过 Cocos `Widget` 固定到安全区左边 60 px、下边 35 px，尺寸 220×220；底盘、操纵帽和触摸事件全部属于该容器。
- 摇杆中心操纵帽：初始位于底盘中心，拖动范围限制在底盘半径内。
- `ActionControlsRoot`：整体尺寸 360×190，通过 `Widget` 固定到安全区右边 30 px、下边 30 px。
- 普通撕咬按钮和冲刺撕咬技能按钮：只使用 `ActionControlsRoot` 内部局部坐标，不再分别对齐屏幕边缘。

## 交互约定

- 摇杆拖动持续输出 `InputCommand.move`，松手后回中并停止移动。
- 玩家仍可向任意二维方向移动，但鱼儿显示与攻击朝向仅保留左、右两种；水平翻转切换朝向，上下移动和松手后保持最近一次水平朝向。
- 普通攻击按钮触发 `InputCommand.basicAttack`，按钮按下和释放均提供视觉反馈。
- 技能按钮触发 `InputCommand.skill`，技能冷却期间显示不可用状态并阻止重复触发。
- 控件全部挂在 `SafeAreaRoot/InputLayer`，不随海底世界缩放；刘海屏、平板和折叠屏通过安全区适配。
- 正式美术资源使用 PNG，禁止用 SVG 或 Graphics 拼凑替代；每个正式资源均须登记提示词版本、加工步骤和 `.meta`。

## 登录弹窗基线

- 登录弹窗属于固定屏幕模态 UI，必须与摇杆、技能 UI 使用一致的屏幕空间语义；显示期间暂停玩家移动与相机跟随。
- IDE 与浏览器共享面板尺寸、输入框尺寸、校验和提交逻辑；平台差异仅限输入适配层。
- 单行用户名输入必须设置 `EditBox.InputMode.SINGLE_LINE`，不得使用默认生成 `<textarea>` 的 `InputMode.ANY`。
- Cocos IDE 原生输入 DOM 必须同时以未聚焦、聚焦、输入、失焦和窗口缩放状态验收；不能只检查输入框背景节点。
- 改变弹窗父节点前必须记录 Canvas 位置、Canvas 锚点、HUD 补偿、相机位置和输入节点世界矩阵，不得通过固定偏移试错。
- IDE 登录弹窗的 Cocos 视觉节点必须保留在 `HudRoot/SafeAreaRoot/InputLayer`；原生单行 input 的 DOM 校准只修正聚焦后的输入元素，不得通过把整套弹窗移到 Canvas 根节点来规避坐标问题。

## 坐标系统基线

- `Canvas` 由 Cocos 按 1280×720 设计分辨率对齐屏幕，使用中心锚点；`WorldRoot` 与 `HudRoot` 作为 Canvas 直接子节点时局部位置均为 `(0,0)`。
- `MainCamera` 与 `WorldRoot` 内的游戏对象使用同一局部世界坐标；相机移动到 `(x,y)` 后，`HudRoot` 同步到 `(x,y)`，从而在游戏相机中保持固定屏幕位置。
- Scene 编辑视图的观察相机只用于查看和编辑场景；它显示的三轴手柄、白色相机框和透视角度不是游戏 `MainCamera` 的最终渲染坐标。
- 原生 EditBox input 属于 DOM 视口坐标，只允许通过 `GameCanvas.getBoundingClientRect()` 从设计坐标换算，不得用 Scene 观察相机、节点世界坐标或 Canvas 场景位置直接定位。

## 修改前检查表

1. 完整读取项目 `AGENTS.md` 和本文档。
2. 列出目标节点从 Canvas 到控件的完整父节点链。
3. 标注每层 `UITransform` 尺寸、锚点、Widget 和运行时位置修改代码。
4. 确认控件使用世界、HUD、Canvas 局部或 DOM 中的哪一种坐标系。
5. 确认 IDE 与浏览器是否走同一实现；若不同，列出共享部分和适配差异。
6. 修改后执行双端人工预览与自动检查，未人工确认时标记“待用户视觉验收”。
