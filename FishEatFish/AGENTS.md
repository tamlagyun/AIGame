项目名称：鲫鱼吃鲤鱼
目录名称：FishEatFish

## 协作规则

- 项目必须分步骤开发，每一步开始前先列出本步计划。
- 用户确认总体方案后，后续步骤可按计划连续实施；每步结束时说明完成内容、验证结果和下一步建议。
- 只修改 `FishEatFish`，不得修改或依赖同级其他游戏项目。
- 开发说明、需求、架构、README、计划和协作文档默认使用中文，代码标识符和平台官方名称可使用英文。

## 需求与版本

- 新增或修改需求时，必须同步更新 `docs/game-requirements.md` 和 `docs/requirements-changelog.md`。
- 需求、应用、配置和存档使用独立版本号，不得混用。
- JSON 配置必须包含 `schemaVersion` 和稳定 ID；存档必须支持版本校验和迁移入口。

## 美术与 UI

- 正式 UI 和美术不得使用 SVG、程序化矢量图形或临时图标拼凑。
- 正式资源必须通过工作区 `.agents/skills/jimeng-ai` 接口生成，或使用已登记且授权明确的外部位图资源。
- 每次调用美术 AI 前，必须向用户展示将被原样使用的完整提示词、提示词版本、模型、尺寸、数量、用途和输出目录，并等待明确确认。
- 用户确认后不得静默修改提示词；修改提示词或重新生成必须创建新版本、写入 `docs/art-prompts.md` 并重新确认。
- 原图、加工步骤、选定结果、授权状态、导入路径和 `.meta` 必须登记在 `docs/art-assets-register.md`。
- API 密钥、生成临时文件和未选定候选图不得提交版本库。

## 架构与平台

- 核心玩法层不得导入 `cc` 或任何平台 SDK。
- 微信、抖音、Android、iOS、HarmonyOS API 只能出现在各自平台适配器内。
- UI 通过视图状态或命令接口连接玩法，不直接修改核心规则。
- Cocos 资源文件与对应 `.meta` 必须一起提交，不得手工复用其他资源的 UUID。

## 项目卫生

- 文件使用 UTF-8 和 LF，文件名默认使用英文。
- 缓存、构建产物、日志、密钥、签名、AI 临时输出和依赖目录不得提交。
- 每个开发步骤结束后运行 `npm test`、`npm run check` 和 `npm run hygiene`。
- 删除生成文件前必须校验绝对路径位于 `FishEatFish` 内，不得清理工作区根目录或其他项目。
